# Дизайн: системный поиск и устранение ошибки DECODING COLUMN "DELIVERY_MODE"

**Дата:** 2026-02-26  
**Контекст:** setki-21, moskit-api (Rust/sqlx), PostgreSQL. Ошибка сохраняется после нескольких правок enum (type_name = "VARCHAR", rename = "self_pickup").  
**Режим:** /brainstorm, все эксперты на вызов. Решение — через логирование и поиск корневой причины, не методом тыка.

---

## 1. Контекст и «пять почему»

### Симптом
```
ERROR OCCURRED WHILE DECODING COLUMN "DELIVERY_MODE": INVALID VALUE "SELF_PICKUP" FOR ENUM DELIVERYMODE
```

### Что уже сделано (без устойчивого результата)
- В `moskit-core`: `DeliveryMode` и `PaymentType` с `#[sqlx(type_name = "VARCHAR")]` и `#[sqlx(rename = "self_pickup")]` и т.д.
- Миграция: колонки `delivery_mode`, `payment_type` — `VARCHAR(50)` DEFAULT `'self_pickup'` / `'postpaid'`.
- Деплой: образ с фиксом выкатывается, на сервере `up -d --no-build api`; ошибка всё равно возвращается.

### Пять почему (по COGNITIVE_CODE — корневая причина)

1. **Почему падает запрос?** — sqlx не может преобразовать значение из колонки `DELIVERY_MODE` в enum `DeliveryMode`.
2. **Почему не может?** — Сообщение говорит: value = `"SELF_PICKUP"`, enum = `DELIVERYMODE`. Мы ожидаем строку `"self_pickup"` (snake_case), а в ошибке фигурирует `"SELF_PICKUP"` (SCREAMING_SNAKE).
3. **Почему там SELF_PICKUP?** — Не проверено. Возможные варианты: (A) в БД на проде реально записано `SELF_PICKUP`; (B) тип колонки на проде — native PG ENUM с такими значениями; (C) драйвер/протокол отдаёт или нормализует значение в верхнем регистре; (D) ошибка sqlx просто форматирует значение в верхнем регистре в сообщении.
4. **Почему мы не знаем точно?** — Нет наблюдаемости: не логируется сырое значение колонки до декодирования и не проверяется фактическая схема и данные на проде.
5. **Почему не делали так сразу?** — Исправляли по гипотезе «VARCHAR + rename» без верификации данных и типа колонки на целевой БД.

**Итог:** Корневая причина не установлена. Нужна диагностика: что реально приходит из БД (байты/строка) и что за тип колонки на проде.

## 3. Подходы (2–3 варианта)

### Подход A: Диагностика → точечный фикс (рекомендуется)

- **Шаги:** (1) Добавить логирование сырого значения колонки при ошибке (или до декода в одном месте). (2) На проде выполнить проверку: тип колонки, примеры значений. (3) По результату: либо нормализовать данные/схему, либо изменить маппинг (например, принять и `SELF_PICKUP`).
- **Плюсы:** Устраняем корневую причину, не угадываем.  
- **Минусы:** Нужен один цикл деплоя/проверки с логами или скриптом на проде.

### Подход B: Устойчивый к регистру декод

- Не менять данные в БД; в Rust принимать и `self_pickup`, и `SELF_PICKUP` (например, кастомный `Decode` или обёртка `String` + парсинг в enum с приведением к нижнему регистру).
- **Плюсы:** Работает при любом регистре в БД.  
- **Минусы:** Маскирует несогласованность данных; без логов при следующей подобной ошибке снова будем гадать.

### Подход C: Уйти от enum в FromRow

- Читать `delivery_mode`/`payment_type` как `String`, в ручном коде маппить в enum (с нормализацией регистра). FromRow для `Dealer` использовать только для «простых» полей или обёрток.
- **Плюсы:** Полный контроль над парсингом и логированием.  
- **Минусы:** Больше кода, дублирование маппинга; при росте таких полей — поддерживать сложнее.

**Рекомендация:** A (диагностика + точечный фикс), с элементом B только если на проде окажется, что значения приходят в верхнем регистре и менять данные нельзя.

## 4. Дизайн по секциям

### 4.1 Наблюдаемость и логирование

**Цель:** При любой ошибке декодирования колонки знать точное значение, которое вернула БД.

- В moskit-api (или в слое, где вызывается запрос за дилером): при возврате ошибки от репозитория/запроса — логировать текст ошибки и, по возможности, идентификатор запроса (например, `dealer_id`).
- Вариант 1 (минимальный): в обработчике `get_dealer` / `update_dealer` при `map_err` логировать `e.to_string()` (уже часть сообщения пользователю) плюс `dealer_id` с уровнем `ERROR`.
- Вариант 2 (глубже): перед использованием `FromRow` для `Dealer` выполнить один диагностический запрос `SELECT delivery_mode::text, payment_type::text FROM dealers WHERE id = $1` и залогировать сырые строки; затем выполнить основной запрос. Включать только по флагу (например, `DEBUG_DELIVERY_MODE=1`) или в тестовой среде.

**Критерий приёмки:** При воспроизведении ошибки в логах видно либо сырое значение колонки, либо однозначную ошибку sqlx с указанием значения.

### 4.2 Верификация на проде (ручной шаг / скрипт)

**Цель:** Установить фактический тип колонок и реальные значения в таблице `dealers`.

- Запрос 1: тип колонки  
  `SELECT data_type, udt_name FROM information_schema.columns WHERE table_name = 'dealers' AND column_name IN ('delivery_mode', 'payment_type');`
- Запрос 2: примеры данных  
  `SELECT id, delivery_mode, payment_type FROM dealers LIMIT 5;`
- Записать результат (и при необходимости hex/байты для `delivery_mode`) в runbook или в тикет. По результату решить: нормализовать данные (UPDATE), изменить тип колонки или изменить только маппинг в коде.

**Критерий приёмки:** Документирован ответ: тип колонки (VARCHAR vs enum), фактические значения (регистр, пробелы).

### 4.3 Стратегия фикса (после диагностики)

- Если в БД **VARCHAR** и значения **lowercase** (`self_pickup`): оставить текущий маппинг; убедиться, что на проде крутится образ с этим маппингом (проверка версии/образа api).
- Если в БД **VARCHAR**, но значения **uppercase** (`SELF_PICKUP`): либо (1) нормализовать данные разовым UPDATE, либо (2) в Rust принимать оба варианта (подход B: case-insensitive маппинг или кастомный Decode).
- Если в БД **native ENUM** с другими именами: привести схему к VARCHAR + миграция данных или явно описать enum в sqlx под реальный тип БД.

**Критерий приёмки:** Один выбранный вариант зафиксирован в плане внедрения; после внедрения сохранение настроек в админке не даёт ошибки.

### 4.4 Регрессия и тесты (Анна)

- Юнит/интеграционный тест: запрос к тестовой БД с `delivery_mode = 'self_pickup'`, `payment_type = 'postpaid'` → успешный декод в `Dealer`.
- Опционально: тест с `SELF_PICKUP` (если выбран подход B), ожидание успешного декода.
- В CI прогонять после изменений в moskit-core/moskit-api.

**Критерий приёмки:** Есть автоматическая проверка декода для хотя бы одного набора значений.
