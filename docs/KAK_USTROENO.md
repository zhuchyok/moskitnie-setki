# Как устроен проект setki-21: локалка и VDS

## Один проект, один код, одна база

- **Проект один и тот же** — репозиторий setki-21. Никаких «разных проектов» нет.
- **Код один и тот же** — правишь только у себя на компьютере (локально). На VDS оказывается **копия** этого кода после деплоя.
- **База данных одна** — PostgreSQL живёт **на VDS** (в docker-compose: сервис `postgres`). Локально своей отдельной базы для setki-21 может не быть вообще; если поднимаешь `docker-compose` у себя — это уже **другая** БД (отдельные данные).

Итого:
- **Локалка** = где ты редактируешь код и откуда запускаешь деплой.
- **VDS (45.10.43.248)** = где крутится сайт, API и **та самая база** с дилерами, заказами, настройками цен. То, что видит пользователь и админка на setki21.ru — всегда с VDS.

## Почему «на сайте не видно изменений»

Потому что сайт показывает то, что **уже выложено на VDS**. Пока ты не сделал деплой:

1. Код на VDS — старый (последний раз скопированный).
2. Контейнер `web` на VDS собран из старого кода.

Ты мог сто раз править файлы у себя — на сайте ничего не поменяется, пока не выполнишь деплой и не обновишь страницу в браузере (лучше Ctrl+Shift+R).

## Как дальше работать

1. **Всё правишь только локально** — в своей папке с setki-21 (например `~/Documents/dev/setki-21`).
2. **Чтобы изменения появились на setki21.ru**, из корня этого же репозитория запускаешь:
   ```bash
   ./scripts/deploy_moskit_api_vds.sh
   ```
   Скрипт: копирует код на VDS, пересобирает образ фронта (web), перезапускает api и web. Базу не трогает.
3. **Открываешь сайт/админку** и делаешь жёсткое обновление (Ctrl+Shift+R), чтобы не смотреть кэш браузера.

Повторяй шаг 2 после любых правок, которые должны попасть на прод.

## Где что лежит

| Что | Где |
|-----|-----|
| Исходный код (единственный) | У тебя на диске: репозиторий setki-21 |
| Копия кода на проде | VDS, каталог из `.env.deploy` (сейчас `/home/atra/app/setki21_src`) |
| База данных (дилеры, заказы, цены) | Только на VDS, в контейнере `postgres` |
| Сайт setki21.ru | Отдаётся с VDS (контейнеры `web` + `api`) |

## Если хочешь тестировать локально с «как на проде»

Поднять у себя полный стек (код тот же, база своя):

```bash
docker-compose up -d
```

Сайт: http://localhost:3003, API: http://localhost:8083. База будет пустая или с тестовыми данными — это **не** база с VDS. Чтобы повторить данные с прода — нужен дамп с VDS и его восстановление локально (отдельная задача).

---

Кратко: **проекты не разные, базы не разные между «двумя средами»** — есть один прод на VDS с одной БД. Локалка — только место для правок и запуска деплоя; после деплоя на VDS оказывается тот же код, и сайт работает с той же базой.
